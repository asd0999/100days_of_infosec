CTFd —  labyrinth (part 1)

Looking at an interesting reverse engineering challenge today. Here’s the graphical disassembly view of the problem in Binary Ninja :

The main function takes user input and calls the traverse function with that input, if the return value from the traverse function is equal to 0x257b (or 9595) then the win condition is met and the flag is retrieved

The traverse function :

The traverse uses 3 local variables :

  1. The user input populated from the RDI register
  2.  A total variable
  3. A variable that points to the .data section of the assembly

I have renamed the variables in the picture above, though it took me a long time to figure what the third variable was pointing towards. The value ‘a’ didn’t make any sense to me. I tried converting it to hex, decimal, maybe it was a character? But characters aren’t represented like this.

Then when I double clicked on it, I landed up at a memory address that contained some data stored in the program

The decompiler view in Ghidra wasn’t of much help either

Looking ahead into traverse though, I can see that the program is comparing the user input with 0x4C or 0x52 which in ASCII represent ‘L’ or ‘R’

So I need to give directions in terms of L and R which is consistent with the prompt the program gives at the beginning but I need to spend more time unravelling what the traverse function is doing. Until tomorrow!
