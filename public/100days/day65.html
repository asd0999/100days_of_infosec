CTFd —  labyrinth (part 2)

Ok, so I sat with the assembly today and went through it line by line and made way more sense of what’s going on. Let’s start with the main function. The main does the following things :

  1. Puts - Print a prompt about needing directions to get out of a maze
  2. Fgets - Asks the user for an input
  3. Strchr - Search for a newline character in the user input
  4. Call traverse () - Send the user input up till the newline character to the traverse function  
  5. Cmp - Compare the return value of the traverse function with 0x257b (0r 9595). If equal, direct to branch “Print flag” [WIN!]

This is what the traverse function is doing :

Every node (a through g) of this .data section has a number (first 2 bytes) and two memory address where the program will jump to depending on the input

If the input is L, is jumps to address at the 8 byte offset

If the input is R, is jumps to address at the 16 byte offset

For example, if I enter L, the program will jump from ‘a’ to the address at the 8 byte offset being 6010b0 (read little endian) which is node ‘c’

At every jump the number associated with the node  is cumulated and returned to the main function to check against the sum == 9595 win condition

Here, ‘a’ has the number 0xe3  and ‘c’ has the number 0x468, the total being 0x54b which is != 0x257b

So we need to find a way to jump to the nodes in way that is :

  1. Legal - in the sense that every node can only jump to two other
  2. Sum - that results in the sum being equal to 0x257b or 9595

I tested this out using gdb and observing the RAX register through the instructions

Next step, I need to write a p=solver that takes these conditions into account and brute forces a solution. I’m guessing there are more than one solution to this problem
