CTFd —  Brutus (part 2 - gdb over socket)

After the program creates the socket, the program listens and accepts for new connection, each of which creates a new child process (see the fork function call) :

The fork is helpful because then we can bombard the server with requests to brute force the canary, else we would have to wait for each connection to be made and broken in order to connect again which would take forever

So the binary given by the challenge is the server binary and the client can connect to it by simply netcat’ing to it on localhost at port 8000. So we need to run the binary in one terminal and connect to it from another terminal

Terminal 1   →   ./new_brutus

Terminal 2   →   nc 127.0.0.1 8000

So remote GDB debugging works as long as the machine running the server binary has GDB

from pwn import *

context.log_level ='debug'

gdb_script = '''

set follow-fork-mode child

break handle

continue

'''

p = gdb.debug('./new_brutus', gdb_script)

sleep(1)

r = remote('127.0.0.1', 8000) #local process but connecting over socket

r.interactive()

NOTE: The set follow-fork-mode child is important because it tells GDB to follow the child process thread instead of the parent process after the program forks  

NOTE: gdb.debug() starts a process as opposed to gdb.attach() attaches to an existing process. The second argument for both are the gdb command
